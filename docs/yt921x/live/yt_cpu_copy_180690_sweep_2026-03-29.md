# YT9215 `0x180690` CPU-copy candidate sweep (2026-03-29)

## Goal
Probe whether `0x180690` controls a copy/trap path that can expose LAN UU/MC
traffic on WAN or CPU conduits.

## Baseline dump (`0x180688..0x1806a0`)
- `0x180688 = 0x0000003f`
- `0x18068c = 0x0000003f`
- `0x180690 = 0x00000001`
- `0x180694 = 0xdeadbeef`
- `0x180698 = 0xdeadbeef`
- `0x18069c = 0xdeadbeef`
- `0x1806a0 = 0x00000000`

## Method
- For each case:
  - write test value to `0x180690`
  - generate UU + multicast from router:
    - UU target: `192.168.2.199` with static neighbor `02:de:ad:be:ef:00`
    - MC target: `239.1.2.3`
  - restore `0x180690` to baseline `0x00000001`
- Simultaneous capture points:
  - Pi WAN side: `eth0` filtered for UU/MC target traffic
  - Router conduits: `eth0`, `eth1` raw capture
- Capture metric recorded: `N packets captured` per interface/log.

## Observed behavior
- WAN-side (`Pi eth0`) stayed `0` for all tested bits (`0..10`).
- Router `eth1` showed low non-zero background/activity in all cases.
- Router `eth0` remained near-zero for most cases, but rose on:
  - `b3`, `b5`, `b7` (small)
  - `b9`, `b10` (larger)

## Interpretation (current confidence)
- No evidence that `0x180690` bit flips (`0..10`) expose the UU/MC probe
  stream onto WAN side in this runtime.
- Bits `9` and `10` likely influence some CPU-side copy/exception path on
  router `eth0`, but this needs targeted packet-signature decoding to separate
  probe traffic from ambient control traffic.
