# YT9215 unknown-reg action probe summary (2026-03-23)

## Source log
- `docs/yt921x/live/yt_180734_180738_action_probe_20260323_204904.txt`

## Test setup
- Router WAN conduit forced to `eth0` during test, restored to `eth1` at end.
- Registers swept:
  - `0x180734`
  - `0x180738`
- CPU selector fields swept per register:
  - `cpu=4` -> bits `[9:8]`
  - `cpu=8` -> bits `[17:16]`
- Action values tested: `0,1,2,3`.
- Traffic checks per case:
  - router -> Pi ping: `ping -I wan -c 3 172.16.9.1`
  - Pi capture: ARP/ICMP/ICMPv6 on `eth0`
  - Pi -> router ping: `ping -c 2 192.168.2.1`

## Baseline and restore
- Baseline before sweep:
  - `0x180734=0x00020000`
  - `0x180738=0x00020000`
- Restored after sweep:
  - `0x180734=0x00020000`
  - `0x180738=0x00020000`
- Also restored: `wan@eth1` conduit.

Notes:
- The only action-sensitive field in this sweep is `0x180734[9:8]`.
- `action=1` and `action=2` at `0x180734[9:8]` block router->Pi traffic in this scenario.

## Capture and dmesg observations
- For PASS cases, Pi capture consistently shows ARP + ICMP request/reply from router traffic.
- For all cases, `Pi -> router ping` (`192.168.2.1`) does not receive replies.
- For all cases, router dmesg keeps showing ingress on `eth1` path:
  - `net eth1: Unexpected EtherType ... ingress=eth1 cpu_dp=8`
- Therefore, this register sweep does **not** change the return-path misrouting symptom (ingress still appears on `eth1/cpu_dp=8`).

## Working interpretation (from this probe only)
- `0x180734[9:8]` behaves like a per-CPU action gate relevant to the active WAN->CPU path.
- `0x180734[17:16]` and all tested fields in `0x180738` showed no functional effect under this test workload.
